# CLAUDE.md

Guidance for contributing to **Longpipe** — an MIT-licensed, browser-native SDK for
real-time video effects (portrait matting → virtual backgrounds / blur) and audio
denoise. Everything runs **in the browser** as WebGPU/WebGL2 shaders + WASM; there is
no server-side inference. The runtime is the moat; effects are the products.

## What this package is

- One published npm package (`longpipe`). Public entry: `src/index.ts`, which exports
  `EffectsPipeline` (internally `Pipeline`) plus types and the low-level `Backend`.
- Two layers (see `README.md` → "How it works"):
  - **`src/model/`** — the ML runtime: a `Backend` abstraction (WebGPU + WebGL2),
    hand-written ops/shaders, the EfficientNet-Lite encoder + U-Net decoder networks,
    a U-Net wrapper, a temporal ConvGRU, and the compositor/effects.
  - **`src/pipeline/`** — browser plumbing: capability/transport detection, the
    **Web Worker** that owns the GPU, autotune, the adaptive controller, background
    normalization, and the public `Pipeline` class.
  - **`src/audio/`** — an **independent** AudioWorklet denoise subsystem (RNNoise /
    DeepFilterNet3). It does NOT run in the video worker and does not block video
    `ready`.

## Commands

```bash
npm run build      # tsup → dist/ (ESM + CJS + .d.ts). Inlines worker & worklet.
npm run typecheck  # tsc --noEmit (strict)
npm test           # vitest run — BROWSER mode (Playwright Chromium). Needs a real GPU.
npm run test:watch
npm run dev        # vite serving the demo/ apps (demo/vite.config.ts)
```

There is no lint step. `npm test` launches a real Chromium via Playwright because every
op test reads back actual GPU output — it will not run headless without GPU access.
`*.local.test.ts` tests need local fixtures under `tests/fixtures/local/` (gitignored)
and are skipped otherwise.

## Conventions that are easy to get wrong

- **Imports use the `~/` alias = `src/`, and ALWAYS include the explicit extension**
  (`.ts`, `.wgsl`, `.glsl`). e.g. `import x from '~/model/backend.ts'`. The alias is
  wired in three places that must stay in sync: `tsconfig.json` (`paths`),
  `vitest.config.ts` (`resolve.alias`), and the `tildeAlias` esbuild plugin in
  `tsup.config.ts`. There is no extension/index resolution — the import site is
  authoritative.
- **Shaders are imported as text.** `.wgsl`/`.glsl` resolve to a default-export string
  (declared in `src/shaders.d.ts`; transformed by tsup's `loader` and vitest's
  `shaderPlugin`). Add new shader extensions to all loaders if you introduce any.
- **Tensors are NHWC, vec4-packed, and channel count is ALWAYS a multiple of 4**
  (`Tensor.c`). Pad channels to 4 at the boundaries. PyTorch `[B,C,H,W]` fixtures are
  converted to this layout by `scripts/generate_fixtures.py` (`to_nhwc`).
- **The `Backend` interface (`src/model/backend.ts`) is the contract.** Both backends
  must implement it identically; ops are output-identical across backends (tests assert
  this, often to ≤1 ULP / `THRESHOLD = 1e-4`). Don't add an op to one backend only.
- BatchNorm is **fused into conv weights at export** — there is no BN op at inference.

## Adding or changing a GPU op (the most common contribution)

An op exists in *both* backends. To add one, touch all of these:

1. `src/model/backend.ts` — add the factory signature to `Backend["ops"]` (or
   `presenters`) and any param interface.
2. **WebGPU**: `src/model/backends/webgpu/ops/<name>.ts` extending `WebGPUOp`
   (`base_webgpu_op.ts`), plus **two** shaders `shaders/<name>.wgsl` and
   `shaders/<name>_f16.wgsl`. f16 is a separate shader file selected by
   `backend.dtype` — see `ops/conv2d.ts` for the pattern. Bind-group order is fixed:
   **inputs → weights → uniforms → output**.
3. **WebGL2**: `src/model/backends/webgl/ops/<name>.ts` extending `WebGLOp`, plus
   `shaders/<name>.glsl`. WebGL packs tensors into textures (see the comments in
   `webgl/ops/conv2d.ts` for the texture layout convention); one GLSL file serves both
   precisions.
4. **Register** the factory in BOTH `webgpu/index.ts` and `webgl/index.ts`.
5. **Test** in `tests/webgpu/<name>.test.ts` and `tests/webgl/<name>.test.ts` against a
   shared fixture in `tests/fixtures/` (generated by `scripts/generate_fixtures.py`,
   which needs PyTorch). Use `tests/helpers/backends.ts` (`createWebGPUBackend`,
   `BACKENDS`, `BACKENDS_DTYPE`) — don't hand-roll backend construction.

Many ops are **fused** (e.g. `Conv2dAdd`, `ProjResidual`, `ConcatConv2d`,
`GatesFused`/`CandUpdateFused`, `UpFinalSkip`) to eliminate intermediate buffers. When
adding a fused op, keep an unfused reference path mentally so you can validate against
composed primitives.

## Weights

- Format spec: **`WEIGHTS_FORMAT.md`** (read it before touching loading). Each op
  extracts its own opaque blob; blocks route named sub-blobs to children; the model
  routes the full nested `ModelWeights` object by name.
- Two serializations, one loader type: JSON (`number[]`, tests) and a binary `.bin`
  pack (4-byte header length + JSON header with `{offset,length}` + packed float32).
  See `src/model/weights.ts` / `src/utils/weights.ts` / `src/utils/loadWeights.ts`.
- **f16:** weights can arrive as `Float32Array` (from `.bin`) or raw `Uint16Array`
  fp16 bits (from `.f16.bin`); `backend.upload()`/`writeView` converts to the backend's
  storage dtype either way (`src/utils/fp16.ts`). The pipeline prefers the `.f16.bin`
  pack and falls back to `.bin` (`fetchModelWeights` in `pipeline/index.ts`).
- **Model variants are real classes, not config flags.** `large` / `small` / `xl` are
  separate network files in `src/model/networks/`; `ModelWeights` reflects this with
  optional fields (`s5?`, `s6?`) and variable-length arrays. A new variant = a new
  network file + a matching type extension, visible as a diff.

## Presets & runtime adaptation

- Five tiers `xs → xl` in `src/pipeline/presets.ts` (`PRESETS`, linear cost order).
  Named presets: `fast=xs`, `balanced=medium`, `quality=xl`. `auto` (default) runs a
  microbench at init (`worker/autotune.ts`, `resolve_preset.ts`) and picks the best fit.
- A preset's `resolution` **must** equal `TIER_CONFIG[model].canvasRes` (the model
  layer is authoritative; `tests/networks/tier_config.test.ts` asserts agreement).
- `skipFrames` = input frames the model sits out; the **compositor still renders every
  frame** using the most recent alpha matte (`RenderOp` in `src/model/render_op.ts`).
- The adaptive controller (`pipeline/adaptive.ts`) only acts when `preset:'auto'` and
  `adaptive` is not disabled. Explicit preset choices are never auto-overridden.
- **Note:** at runtime the worker always uses f16 when the device supports `shader-f16`,
  regardless of a preset's listed `dtype` (see `handleInit` comment) — bench timings can
  underestimate large/xl cost on f32-only devices. Acceptable for v0.1.

## Pipeline / worker model (gotchas)

- The video model runs **inside a Web Worker**. `Pipeline` (main thread) is a thin
  coordinator talking to it over `WorkerController` (request/response + events).
- **Two-phase init handshake:** `init` (no weights — sets up backend, starts the pipe in
  passthrough, runs autotune) → main thread fetches weights for the resolved preset →
  `startRender` (weights) attaches the network in place and emits `ready`. Passthrough
  video flows the whole time; there is no pipe restart.
- **`pipeline.stream` is available synchronously** and emits unprocessed input until the
  effect is live (`~1–3s` cold). `pipeline.ready` resolves on the **video** first frame;
  it does NOT wait on audio.
- **Worker is spawned from a Blob URL** built from `WORKER_SOURCE`, which tsup's
  `inlineWorker` plugin fills with the bundled worker at build time. In dev (`vite`),
  `WORKER_SOURCE` is empty and it falls back to a `new URL(...)` module worker. Same for
  the AudioWorklet (`WORKLET_SOURCE` / `inlineWorklet`) — but the worklet has **no ESM
  support**, so it MUST be pre-bundled (no dev URL fallback). Don't break these plugins.
- `Pipeline` is a custom **thenable** (`await pipeline` → the instance). `ready` is
  `Promise<void>` on purpose to avoid a thenable-follow cycle — see the long comment on
  the `ready` field. Don't "simplify" it to `Promise<this>`.
- Per-browser frame transport is auto-selected (`pipeline/topology.ts`,
  `transports/`): `MediaStreamTrackProcessor`, `transferControlToOffscreen` +
  `captureStream`, or an `ImageBitmap` shuttle. All invisible to callers.
- Errors: async/runtime errors go through the `onError` callback (and reject `ready`
  if they happen during init); **synchronous constructor errors throw out of
  `new Pipeline()`** and do not hit `onError`.

## Tests

- Browser-mode vitest; `tests/webgpu/` and `tests/webgl/` mirror each op. `tests/blocks/`
  and `tests/networks/` run dual-backend via `describe.each(BACKENDS)`.
- Numeric tolerance is tight (`1e-4` for f32 ops). f16 paths use the `BACKENDS_DTYPE`
  matrix and should self-skip a row when the backend/feature isn't available.
- Fixtures are PyTorch-generated and **gitignored** (`tests/fixtures/**.json`). If a
  test can't find its fixture, regenerate with `scripts/generate_fixtures.py`.

## Caveats / loose ends to be aware of

- Several source comments reference docs that **do not exist in this repo**:
  `docs/PIPELINE.md`, `docs/MODEL_PLAN.md`, `project_backend_canvas_contract.md`. Don't
  send readers there; the behavior is in the code + this file. (Fixing/porting those
  docs is a welcome contribution.)
- Training scripts, the weight export pipeline, and fixture generation live outside this
  SDK repo (referenced as `training/`, `sdk/scripts/`); only `scripts/generate_fixtures.py`
  and `scripts/generate_weights_index.py` ship here.
- Default weights are fetched from `https://cdn.longpipe.dev/models/v/0.0.4/` (versioned
  path, set in `pipeline/index.ts`). Self-hosting: mirror the `model_<name>.bin` files
  and pass `weightsBaseUrl`.
- Project is pre-1.0 (`0.0.x`): the public API is explicitly expected to change between
  versions. Keep the public surface in `src/index.ts` deliberate and minimal.

## Style

Match the surrounding code: TypeScript strict mode, two-space indent, explicit import
extensions, dense explanatory comments on non-obvious GPU/lifecycle decisions (the
codebase favors a comment explaining *why* over terse code). New ops/files should read
like the existing ones in the same directory.
