# Longpipe — Roadmap Execution Plan

An execution plan for an autonomous coding agent (Opus 4.8) to deliver the
[product roadmap](https://longpipe.dev) in a sequenced, dependency-aware, and
**verifiably cross-browser/cross-device** manner.

The roadmap's own thesis is the lens for every decision: **the runtime is the moat;
effects are applications of it.** So the plan front-loads the runtime/test foundation
that makes every later effect cheap and safe to ship, and treats new web APIs as
*upgrades absorbed behind a stable seam, never rewrites*.

> **Depth gradient (mirrors the memo's own stance).** This plan goes deep on **Now**
> (Phases 0–3), medium on **Next** (Phases 4–7), and seams-only on **Later** (Phase 8) —
> per the memo: *"build the abstraction so you're already-correct when it flips; don't
> bet the timeline on it."* Re-plan Next/Later in detail when Now is green.

---

## 0. Operating model — how the agent executes each unit

Every work item is a **PR-sized unit** delivered through the same loop. This is the
non-negotiable definition of "done well."

1. **Branch** off `main` (never commit to `main` directly).
2. **Scope** to one unit. If a unit needs >~400 lines or touches >1 subsystem, split it.
3. **Implement** matching surrounding code (see `CLAUDE.md` → Style). Dual-backend
   changes land in the *same* PR — never one backend ahead of the other.
4. **Test** at the lowest meaningful level: op fixture → block → network → pipeline.
   New behavior ships with new tests in the same PR.
5. **Verify parity**: WebGPU and WebGL2 outputs agree within tolerance (`≤1e-4` f32).
6. **Verify budget**: no per-frame regression on the benchmark harness (Phase 0.4).
7. **Verify matrix**: the cross-browser/device matrix (Phase 0.2) is green, or the gap
   is explicitly documented as a known-limitation with a fallback path.
8. **PR** with: what/why, test evidence, matrix result, perf delta, and rollback note.
9. **Update docs**: `CLAUDE.md`, `README.md`, and the relevant `docs/*.md` in the same PR.

**Definition of Done (every unit):** typecheck clean · new+existing tests green on
WebGPU & WebGL2 · parity gate green · perf budget green · matrix green or documented ·
public API change is intentional and typed · docs updated.

**Guardrails (the anti-roadmap — refuse work that violates these):**
- ❌ Heavy AR / face filters / makeup (perf moat doesn't transfer).
- ❌ Becoming a video product / hosted compositing as the *core* (kills the in-browser moat).
- ❌ Walling any effect behind a license — everything in `src/` stays MIT.
- ❌ Server-side rendering as the primary path.

## Phase 0 — Engineering foundations *(blocks everything; do first)*

**Goal:** make "well-tested across devices/browsers" a property the CI *enforces*, not a
claim. Today there is **no CI, no lint, no device matrix, no regression benchmark, no
soak test**. Nothing else in the roadmap is trustworthy until this exists.

| # | Unit | Detail | Acceptance criteria |
|---|------|--------|---------------------|
| 0.1 | **CI pipeline** | GitHub Actions: `typecheck` → `build` → `test` (Playwright Chromium, WebGPU+WebGL) on every PR. Cache npm + Playwright browsers. | PR cannot merge unless green. Build artifact (`dist/`) size tracked and diffed. |
| 0.2 | **Cross-browser/device matrix** | Add Playwright projects for Chromium, Firefox, WebKit. Add a real-device cloud lane (BrowserStack/LambdaTest or self-hosted lab) for **iOS Safari, Android Chrome, Safari macOS** — emulation does NOT reproduce GPU/driver behavior. Tag matrix-only tests. | A documented support matrix (`docs/BROWSER_MATRIX.md`) with green/known-gap per cell, regenerated by CI. |
| 0.3 | **Capability & forced-fallback harness** | Test seams that force each rung of `WebGPU → WebGL2 → WASM`: mock `navigator.gpu` absent, force `shader-f16` off, simulate `device.lost` / `webglcontextlost`. Drives Phase 1.1. | Each fallback rung has an automated test proving the pipeline keeps emitting frames. |
| 0.4 | **Benchmark + regression harness** | Promote `demo/benchmark.*` into a headless, reproducible harness: per-tier model-ms, FPS, alpha **MAE/IoU** vs reference, on a fixed input clip. Stores a baseline; CI fails on >X% regression. Doubles as the public "vs MediaPipe" asset (Phase 3.2). Must measure **single vs stacked effect budgets** and CPU% vs MediaPipe per the [performance model](PERFORMANCE_MODEL.md) §5. | `npm run bench` produces a stable JSON report; CI diffs against committed baseline. |
| 0.5 | **Soak / memory-leak harness** | Long-session run (e.g. 30–60 min equivalent, or N-thousand frames) asserting bounded GPU buffer count + JS heap. Catches the "long-session memory stability" roadmap item. | Soak test in CI nightly; flat memory curve within tolerance. |
| 0.6 | **Lint + format + pre-commit** | ESLint (typescript-eslint) + Prettier matching current style; `engines` field; optional Husky pre-commit (`typecheck` + changed-file lint). | `npm run lint` clean; CI enforces. |
| 0.7 | **Golden/parity test scaffolding** | Generalize `tests/helpers/backends.ts` into a parity runner: any op/network asserted WebGPU≡WebGL within tolerance from one fixture. Visual-regression snapshots for compositor output. | Adding an op auto-gets a parity test via the helper; snapshot diffs gate PRs. |

**Exit criteria for Phase 0:** a PR that changes a shader cannot merge unless it is
proven correct and non-regressing on the full backend × browser matrix, automatically.

> obtain real-GPU environments, deterministic synthetic-media harness, tolerance strategy, CI
> tiering, and build order) lives in [TESTING_STRATEGY.md](TESTING_STRATEGY.md).

internal seam for everything in Next. Depends on Phase 0's harnesses to be provable.

| # | Unit | Files | Acceptance criteria |
|---|------|-------|---------------------|

## Phase 4 — WebCodecs path (roadmap: *Next / Runtime*) — *absorb as an upgrade*

**Goal:** add the encoded-stream / efficient capture+recording path *behind the existing
transport seam* (`pipeline/transports/`, `worker/adapters/`) so it's a new rung, not a
rewrite. Gate on the same parity + matrix + budget. Prereq: Phase 1 + WebCodecs support
probe in the capability ladder (Safari 26 ships it; keep WASM/canvas fallback intact).

## Phase 5 — Pluggable model layer (roadmap: *Next / Runtime*)

**Goal:** let a caller swap the matting network for another `ModelRunner` (Phase 1.6's
seam) without touching the pipeline. Enables custom/per-brand models (a *Later* effect)
and third-party experimentation. Ship with a conformance test kit that any runner must
pass (shape/parity/budget).

---

## Phase 6 — On-device captioning flagship (roadmap: *Next / Effects*)

**Goal:** the "this is a runtime, not a blurrer" proof. A streaming on-device ASR effect
riding the *same* runtime patterns (worker, weights pipeline, autotune, telemetry).
Highest new-subsystem risk in Next — scope it as its own mini Phase 0→3 (foundations →
model → DX). Privacy story ("media never leaves the browser") is the headline.

---

## Phase 7 — Telemetry & first-contract enablement (roadmap: *Next / Trust & Commercial*)

**Goal:** an **opt-in** quality-telemetry harness (FPS / mask quality / fallback rung /
zero-copy counters from 1.3) that is both an SLA/sales asset and a field-debugging tool.
Opt-in by default, no media leaves the device, fully MIT. Feeds the "who's using" surface
and the support/integration contracts.

## Phase 8 — Later horizon (seams only; do **not** bet the timeline)

Per the memo, these are *frontier*: build the abstraction so Longpipe is already-correct
when the platform flips, and ship nothing speculative on the critical path.

- **WebNN / NPU** — implement as a third `Backend`/`ModelRunner` behind Phase 1.6's seam,
  guarded by a capability probe. Chrome Origin Trial only today; no Safari/Firefox, no
  Android. Keep WebGPU as the durable production core.
- **Mobile/Safari WebGPU frontier · React Native / WebView story** — track as matrix
  cells (0.2) that flip green as platforms mature.
- **Hosted convenience layer** (model CDN, dashboard, licensing) — the *one* allowed
  paid-operation surface; SDK + runtime stay 100% open (the memo's recommended fork).
  Out of this repo's core; design the client seam so self-host and hosted are identical.
- **Effects depth** — chroma key, multi-person framing, virtual lighting, avatar/relight
  primitive. Each is "just another effect" *because* Phases 0–5 made effects cheap.

## Cross-cutting standards (apply to every phase)

**Testing pyramid.** op fixture (vs PyTorch, `≤1e-4`) → block → network → pipeline →
matrix/soak/bench. Every layer in CI; the expensive layers (real-device, soak) nightly.

**Backend parity is a gate, not a goal.** No op/effect merges WebGPU-only. The parity
runner (0.7) is the enforcement.

**Performance budgets.** Each tier has a committed per-frame ms budget; the bench (0.4)
fails PRs that regress it. New effects declare their budget before merge.

**API stability.** The public surface (`src/index.ts`) is curated; pre-1.0 changes are
deliberate, typed, and changelogged. Internal refactors (e.g. 1.6) must not change public
types without a note.

**Web-API timing discipline.** WebGPU/WebGL2/WASM = durable core (bet here). WebCodecs =
adopt in Next behind the seam. WebNN = seam-only until it actually ships broadly. Every
new API enters through a capability probe with an intact fallback rung — *upgrade, never
rewrite.*

**Privacy by construction.** Media stays in the browser. Telemetry is opt-in and
media-free. This is both the product promise and a commercial differentiator — never
regress it.

## Appendix A — Build vs. adopt: model sourcing *(licenses verified June 2026)*

For an in-browser, real-time, **MIT-redistributable** SDK, a candidate model must pass a
triple filter: **(1) permissive license (incl. *weights*) × (2) small/fast enough for
real-time WebGPU/WebGL on the netbook→phone range × (3) portable into the op set, or a
WASM runtime for audio/ASR.** That intersection is *narrow for visual perception* and
*wide for audio/ASR* — which is why the visual core must be trained (the moat) while
audio and captioning can be adopted.

### Verified candidate table

| Candidate | Use-case | License (verified) | Real-time in-browser? | Verdict |
|---|---|---|---|---|
| **Whisper** (OpenAI) | Captioning / ASR | **MIT** (code + weights) | Yes — ONNX via transformers.js, WebGPU + WASM fallback | 🟢 **Adopt** (batch/near-live) |
| **Moonshine** (Useful Sensors) | Streaming captioning | **MIT** (code + weights), `moonshine-js` on npm | Yes — built for low-latency on-device streaming | 🟢 **Adopt** (live captions) |
| **DeepFilterNet3** | Audio denoise | **MIT/Apache dual**, weights permissive | Yes — already shipping (`dfn`/`dfnint8`) | 🟢 **In use** |
| **RNNoise** | Audio denoise (fallback) | BSD | Yes — already shipping | 🟢 **In use** |
| **BiRefNet** | Matting *teacher* | **MIT** | No (too heavy for live) — distillation teacher only | 🟢 **Teacher only** (in use) |
| **MODNet** (ZHKKKe) | Portrait matting | **Apache-2.0** (code + weights) | Marginal in-browser; quality below the target bar | 🟡 **License-OK, but you out-train it** |
| **MediaPipe Selfie / BlazeFace** | Seg / face-detect | Apache-2.0 | Yes | 🟡 **OK for auto-framing**, not the matting hero |
| **RVM** (Robust Video Matting) | Video matting | **GPL-3.0** ❌ | Yes, but license forbids MIT redistribution | 🔴 **Disqualified (license)** |
| **briaai RMBG-1.4 / 2.0** | Background removal | **Non-commercial / Bria license** ❌ | — | 🔴 **License trap — looks open, isn't** |
| **Zero-DCE / Zero-DCE++** | Low-light | **CC BY-NC 4.0** ❌ | Tiny + fast, but non-commercial | 🔴 **Disqualified (license)** |
| **DPR / PortraitRelighting / IC-Light / SwitchLight** | Relighting | Mixed (Apache / academic / commercial) | No — NeRF/diffusion, not real-time-in-browser | 🔴 **Custom / research / defer** |
| **RTGaze & gaze-redirection research** | Eye-contact | Academic, thin | No production in-browser path | 🔴 **Custom / research / defer** |

### What this means for the roadmap

**Must train (your pipeline) — the moat:**
1. **Matting family** — no permissive model clears real-time-in-browser at your quality bar
   (RVM = GPL, RMBG = non-commercial, BiRefNet = too heavy, MODNet = below bar). You already
   train this; keep investing. ✅
2. **Eye-contact** and **virtual relighting** — no real-time, in-browser, permissive option
   exists. Treat as research spikes; do **not** commit them to a dated Next milestone.
3. **Per-brand matting** (Later) — fine-tune the existing model, not from scratch.

**Adopt permissive OSS — no training:**
- **Captioning** → Whisper (MIT) and/or Moonshine (MIT). The flagship needs *no* model
  creation — the work is streaming + in-browser perf. This de-risks it sharply.
- **Audio denoise** → already done with permissive models.
- **Auto-framing** → BlazeFace (Apache) *or* derive a bbox from the matte you already compute.

**No model at all (shader/DSP/compositing):** all backgrounds, chroma key, overlays,
basic low-light (exposure/tone-curve shader — *the path forward since Zero-DCE is
non-commercial*), basic touch-up (mask-guided surface blur), audio depth.

### Cross-cutting caveats (carry into any "adopt" decision)
- **"Adopt" ≠ drop-in.** A visual model must be re-expressed in the op set + `WEIGHTS_FORMAT.md`
  to run in the zero-copy GPU pipeline; ASR runs via a separate WASM/ONNX runtime, not the
  shader ops. Budget porting/distillation, not just download.
- **Check the *weights* license separately from the code license** — RMBG and several
  matting/relight repos ship permissive code with restricted weights. Re-verify at adopt
  time; the landscape moves fast.

## Appendix B — Research spikes (eye-contact, relighting)

Two roadmap effects have **no real-time, in-browser, permissively-licensed model** today
(Appendix A). They are *not* committed deliverables. They are **time-boxed research
spikes** that must earn their way onto the roadmap by passing an explicit go/no-go bar.
A spike's output is a **decision + evidence**, not shippable code. Treat shippable code as
a bonus, never the goal.

### Spike rules

- **Timeboxed and hard-stopped.** When the box is spent, you **stop** and write the memo —
  even mid-promising. Extensions require a new spike with a new box, justified by the memo.
- **One owner, isolated branch.** Never on the critical path; never blocks a Now/Next unit.
- **Kill by default.** If the kill-criteria are hit *or* the bar isn't *clearly* met, the
  answer is **no-go**. Ambiguous = no-go. The expensive failure mode is a half-built effect
  that ships at 22 fps with artifacts and dents the quality reputation the matting moat earns.
- **Definition of done = the memo**, containing: the prototype's measured numbers, the
  go/no-go call, the licensing finding, and — if no-go — the *condition that would reopen it*
  (e.g. "revisit when a WebGPU-portable, MIT gaze model with ≥X quality appears").
- **Respect the anti-roadmap.** Neither spike may drift into general face-filter / makeup AR.
  Eye-contact and relighting are *utility* corrections of a real signal, not beautification.

### Spike template (copy per spike)

```
Spike: <name>            Owner: <one person>     Timebox: <N> engineer-days (hard stop)
Question (falsifiable):  Can <effect> run in-browser at <fps target> on <device floor>,
                         at <quality bar>, under an MIT-compatible license?
Prototype plan:          smallest end-to-end test that answers it (offline → WASM/WebGPU)
Go bar (ALL must hold):  perf <…> · quality <…> · license <MIT-compat, weights too> ·
                         artifact rate <…> · portable to op set or WASM runtime
Kill criteria (ANY):     no permissive weights · <fps floor> impossible on mid-range ·
                         visible artifacts on standard test clip · needs depth/hardware
                         the browser can't supply · only NeRF/diffusion-class compute works
Deliverable:             1-page memo: numbers, go/no-go, license note, reopen-condition
```

## Appendix C — Audio use-case coverage *(verified June 2026)*

The roadmap under-specifies audio relative to video (it's essentially *denoise* +
*captioning* + an undefined *"audio depth"*). Audio passes the **same triple filter** as
video, with two browser-specific twists: (1) audio models tend to be **small and
permissively licensed** (the *adopt* lane is healthy), but (2) the **AudioWorklet runs on
the audio render thread against a ~2.7 ms / 128-sample @ 48 kHz deadline, and the worklet
has no ESM** — every model must be pre-bundled (like the existing `WORKLET_SOURCE`). That
budget is the whole feasibility story: Silero VAD (<1 ms) fits; diffusion restoration does
not. The audio subsystem is a separate `AudioWorklet` chain (`src/audio/`) independent of
the video worker, so new audio effects compose cleanly as additional nodes.

### Coverage table

| Use-case | Status | Disposition | Notes |
|---|---|---|---|
| Noise suppression / denoise | ✅ Shipped | Adopt — RNNoise (BSD), DeepFilterNet3 (MIT/Apache) | License-clean |
| Denoise ⊥ browser AEC/AGC | ✅ Phase 2.4 | Coexistence test | Covered |
| Captioning / ASR | ✅ Phase 6 | Adopt — Whisper / Moonshine (MIT) | Covered |
| **VAD (voice activity)** | ➕ **New unit 6.0** | Adopt — **Silero VAD (MIT)**, 1.8 MB, <1 ms/chunk, browser-proven | *Prerequisite to captioning*; also unlocks smart-mute / "you're muted" UX and music-bypass |
| AEC / AGC | 📌 **Stated stance** | **Rely on the browser by design** | Don't rebuild echo cancellation; document the dependency rather than leave it implicit |
| Leveling / EQ / de-ess / compression | ➕ No-model | **DSP in the worklet** | Cheap wins (cf. DaVinci Dialogue Leveler); fold into the no-model effects list |
| Auto-ducking (music under speech) | ➕ No-model | DSP (needs VAD) | Editor-grade nicety |
| **"Audio depth"** (roadmap term) | ⚠️ **Decision point** | depends on definition | **Needs a product definition** — dereverb? spatialization? voice richness/BWE? Each is a different effort (DSP vs. spike) |
| Voice isolation / target-speaker | 🧪 Spike **B.3** | Research (kill-by-default) | Real-time-in-browser unproven; needs enrollment |
| Voice enhancement / bandwidth-extension | 🧪 Spike **B.4** | tier (a) DSP / tier (b) research | "Studio voice"; diffusion/GAN too heavy for the worklet |
| Music / non-speech robustness | ➕ No-model | VAD-gated bypass | Speech-trained denoise damages music — bypass when no speech detected |
| Translation / diarization | 🚫 Likely out of scope | — | Closer to a transcription *product* than an effects runtime; flag explicitly before committing |

### What to add to the plan

- **New unit 6.0 — VAD (Silero, MIT):** lands *with/before* captioning (Phase 6), gates
  segmentation/endpointing, and is reused for smart-mute UX + music-bypass. Same Phase 0
  gates (matrix, perf budget).
- **State the AEC/AGC stance** in `docs/PIPELINE.md`: Longpipe relies on the browser's
  `getUserMedia` echo-cancel/auto-gain by design; the denoiser is built to coexist (2.4),
  not replace them.
- **No-model audio effects** (leveling/EQ/de-ess/auto-duck) join the Phase 2 "effects are
  cheap" list — DSP nodes in the existing worklet chain, no weights, no training.
- **Decision required — define "audio depth"** before it can be scheduled (see Open
  Decisions). Until defined, it is *not* plannable.
