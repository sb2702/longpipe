# Weights Format

## Design principle

Each op receives an opaque blob and extracts its own internals. Blocks receive a blob with one named sub-blob per child op. The model constructor receives the full nested object and routes by name. No caller needs to know the internal structure of its children.

This mirrors the pattern in websr (`src/layers/anime4k/conv2d-3x4.ts`).

## TypeScript types

```ts
// Primitive op weight blobs
interface Conv2DWeights    { weights: number[]; bias: number[] }
interface DepthwiseWeights { weights: number[]; bias: number[] }

// Block weight blobs — one key per child op, named by role
interface MBConvWeights {
  expand: Conv2DWeights
  dw:     DepthwiseWeights
  proj:   Conv2DWeights
}

interface DecoderBlockWeights {
  conv1: Conv2DWeights
  conv2: Conv2DWeights
}

// Full model — explicit structure, variants differ as real code changes
// The BASE matting network's weights — one field per architectural piece.
interface ModelWeights {
  encoder: {
    stem: Conv2DWeights
    s0:   { dw: DepthwiseWeights; pw: Conv2DWeights }  // stage 0 is DW+PW, not MBConv
    s1:   MBConvWeights[]
    s2:   MBConvWeights[]
    s3:   MBConvWeights[]
    s4:   MBConvWeights[]
    s5?:  MBConvWeights[]   // full encoder only
    s6?:  MBConvWeights[]   // full encoder only
  }
  bottleneck: Conv2DWeights
  decoder: {
    blocks:       DecoderBlockWeights[]  // length: 3 for full encoder, 2 for small
    finalUpsample: Conv2DWeights
    outputConv:    Conv2DWeights
  }
}
```

### Tier `.bin` top-level shape

A production tier `.bin` (`model_<tier>.bin`, exported by
`training/deploy/export_sdk_weights.py`) is a composite:

```ts
{
  base:    ModelWeights          // the base matting network above
  wrapper: UNetWrapperWeights    // the hi-res U-Net wrapper (variant lives in tier_config, not here)
  flow?:   FlowWeights           // optical-flow temporal head (rides the base encoder's taps)
  face?:   FaceWeights           // 5-pt face-keypoint decoder (drives landmarks / touch-up)
}
```

The `.bin` carries **only weights** — which base network, wrapper variant, and
resolutions to build is decided by `sdk/src/model/tier_config.ts` (model-as-code).

Two more weight files ship alongside the tiers:

- `model_landmark_mesh.bin` — the 478-point face-landmark regressor
  (`LandmarkWeights`: `{ stem, blocks[10], headConv, fc }`). One model shared by
  all tiers; it consumes a 256² face crop, not tier features.
- `face_topology.json` + `weight_mask.png` — static touch-up assets (canonical
  face mesh + per-region skin-smoothing weight mask), not network weights.
```

## How ops and blocks use weights

Each op calls `backend.upload()` on its own arrays internally. Callers just pass the blob:

```ts
// Conv2D receives { weights, bias } and uploads both internally
class Conv2DWebGPU {
  constructor(backend, input, w: Conv2DWeights, params) {
    const weightsBuf = backend.upload(new Float32Array(w.weights))
    const biasBuf    = backend.upload(new Float32Array(w.bias))
    ...
  }
}

// MBConv receives { expand, dw, proj } and passes each to the right op
class MBConv {
  constructor(backend, input, w: MBConvWeights, params) {
    this.expandOp = backend.ops.Conv2d(input, w.expand, params.expand)
    this.dwOp     = backend.ops.DepthwiseConv2d(..., w.dw, params.dw)
    this.projOp   = backend.ops.Conv2d(..., w.proj, params.proj)
  }
}

// Model routes by name
const model = new MattingModel(backend, frame, weights)
// internally: new MBConv(backend, ..., weights.encoder.s1[0], ...)
```

## Serialization

### Test time — JSON

`number[]` arrays, human-readable. Generated by `sdk/scripts/generate_fixtures.py` for op/block tests and `training/export_model.py` for full model tests. Reference activations at checkpoints are stored as a sibling key or file.

```json
{
  "encoder": {
    "s1": [
      {
        "expand": { "weights": [0.1, ...], "bias": [0.0, ...] },
        "dw":     { "weights": [...],      "bias": [...] },
        "proj":   { "weights": [...],      "bias": [...] }
      }
    ]
  }
}
```

### Production — binary pack

Single `.bin` file fetched once. Format:
- 4 bytes: length of JSON header (little-endian uint32)
- N bytes: JSON header — same nested structure as above, but `number[]` replaced by `{ offset: number, length: number }`
- Remaining bytes: all float32 arrays packed contiguously

Deserializer slices `Float32Array` views from the `ArrayBuffer` by offset and produces an object of the same shape. An optional top-level `__dtype__: "f16"` header tag marks half-precision packs (`model_<tier>.f16.bin`, ~half the download); their leaves become `Uint16Array` views of raw binary16 bits, decoded by the backend on upload.

### Loader interface

Both paths return the same type so the model constructor is agnostic:

```ts
function loadWeightsFromJSON(json: any): ModelWeights    // for tests
function loadWeightsFromBinary(buf: ArrayBuffer): ModelWeights  // for production
```

## Model variants

Different base networks (Small / Large / XL) differ in encoder depth and decoder width. These are real code changes — separate network classes, not config flags. `ModelWeights` reflects this with optional fields (`s5?`, `s6?`) and variable-length arrays (`decoder.blocks`). Adding a new variant means a new network file and a matching type extension, visible as a git diff. The five production tiers map onto three base networks via `tier_config.ts` (xs/small → Small, medium/large → Large, xl → XL).
